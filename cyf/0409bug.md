## 3. 根因分析

### 3.1 直接原因：不安全的文件重写模式

`task_executor.py` 中有两个后处理函数在写回 `merged_authors.jsonl` 时使用了**先截断后逐条写入**的模式：

**位置 1：`_enrich_unknown_affiliations()` — 第 990-993 行**

```python
# Write back
if fixed_total > 0:
    with open(merged_file, "w", encoding="utf-8") as f:  # ← "w" 模式立即截断文件!
        for record in lines:
            f.write(_json.dumps(record, ensure_ascii=False) + "\n")
```

**位置 2：`_validate_and_fix_records()` — 第 1140-1143 行**

```python
# Write back
if fixed > 0:
    with open(merged_file, "w", encoding="utf-8") as f:  # ← "w" 模式立即截断文件!
        for record in lines:
            f.write(_json.dumps(record, ensure_ascii=False) + "\n")
```

这个模式的致命缺陷：

1. `open(file, "w")` **在打开的瞬间**就将文件清空为 0 字节
2. 然后通过 `for record in lines` 循环逐条写入
3. 如果在写入过程中发生任何异常（内存错误、序列化失败、磁盘错误、进程中断等），**已截断的原始数据无法恢复**
4. 文件将停留在部分写入状态（本例中：9 条记录）

### 3.2 异常触发点

`_enrich_unknown_affiliations()` 的写回循环虽然在 `try/except` 内部（第 998-999 行捕获异常），但 `_validate_and_fix_records()` 的写回循环 **没有 try/except 保护**（第 1140-1143 行在 try 块之外）。

在处理 968 条记录的 JSON 序列化时，任何一条记录中出现不可序列化的数据（如 NaN、特殊字符、循环引用等），都会导致 `json.dumps()` 抛出异常。由于已写入 9 条后崩溃，异常向上传播，管线终止，后续 Phase 3-5 均未执行。

### 3.3 可能的异常来源

968 篇论文的数据来自多种外部来源（Semantic Scholar API、Google Scholar 爬取、PDF 解析、LLM 调用），数据质量参差不齐。可能的异常触发器包括：

- 某条记录中包含 `float('nan')` 或 `float('inf')`（`json.dumps` 默认不接受）
- 某条记录中包含不可序列化的 Python 对象
- 某条记录中的字符串包含导致编码错误的特殊字符
- LLM 返回的异常格式数据未被正确清洗
- 在后处理遍历 `lines` 列表时，内存中的数据已被部分修改但未完成

关键差异：**之前所有运行仅处理 14-15 篇论文，而本次运行处理 968 篇**（ViT-22B 是一篇高引论文）。规模差距达 **64 倍**。

具体来说：

1. **边缘数据概率**：14 篇论文中每篇数据都出问题的概率极低；968 篇论文中只要有 1 篇数据异常就会触发 bug。假设单篇论文数据异常概率为 1%，则：
   - 15 篇全部正常的概率：(0.99)^15 ≈ 86%
   - 968 篇全部正常的概率：(0.99)^968 ≈ 0.006% — **几乎必然触发**

2. **数据来源多样性**：968 篇论文来自更多样化的期刊、会议和预印本平台，数据格式和质量变异性远大于 15 篇论文

3. **LLM 调用次数**：后处理中 LLM 调用次数与论文数成正比，968 次调用中出现一次异常返回的概率远高于 15 次

4. **处理时间**：本次运行从 10:40 到 13:24 耗时近 3 小时，长时间运行更容易遭遇临时性问题（网络超时、API 限流、内存压力等）

**结论**：bug 始终存在于代码中，只是之前的小规模运行恰好没有触发。规模扩大后，这个潜在缺陷被暴露出来。

## 5. 解决方案

### 5.1 核心修复：原子写入（Atomic Write）

将两处不安全的文件重写替换为原子写入模式：

```python
import tempfile
import os

# 修复后的写入模式
if fixed > 0:
    # 1. 先写入临时文件
    tmp_fd, tmp_path = tempfile.mkstemp(
        dir=merged_file.parent,
        suffix=".tmp",
        prefix="merged_authors_"
    )
    try:
        with os.fdopen(tmp_fd, "w", encoding="utf-8") as f:
            for record in lines:
                f.write(_json.dumps(record, ensure_ascii=False) + "\n")
        # 2. 写入成功后，原子替换原文件
        os.replace(tmp_path, str(merged_file))
    except Exception:
        # 写入失败，删除临时文件，原文件保持不变
        os.unlink(tmp_path)
        raise
```

**原理**：`os.replace()` 在 Linux 上是原子操作（基于 `rename` 系统调用），要么完全成功替换，要么完全不变。即使写入临时文件时发生异常，原始的 `merged_authors.jsonl` 仍然完整保留。

### 5.2 需要修改的位置

| 文件 | 函数 | 行号 | 说明 |
|------|------|------|------|
| `task_executor.py` | `_enrich_unknown_affiliations()` | 990-993 | 替换为原子写入 |
| `task_executor.py` | `_validate_and_fix_records()` | 1140-1143 | 替换为原子写入 |

### 5.3 额外建议

1. **添加 JSON 序列化预检**：在写入前对每条记录调用 `json.dumps()` 检查是否可序列化，跳过异常记录而非中断整个流程

2. **写入前备份**：在后处理前创建 `merged_authors.jsonl.bak`，即使原子写入也失败，仍有备份可恢复

3. **添加记录数校验**：写入完成后检查新文件行数是否与原始 `lines` 列表长度一致

4. **全面排查类似模式**：检查代码库中所有使用 `open(file, "w")` 重写已有重要数据文件的位置
