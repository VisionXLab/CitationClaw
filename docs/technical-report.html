00 — V2 真实变更摘要

这次 V2 到底改了什么？

以下内容按当前 v2.0.0 分支源码核对。V2 的核心不是版本号或页面文案变化，而是把 v1 中偏 LLM 搜索驱动的流程，重构为结构化数据优先、PDF 证据增强、Skill 调度和可分享报告的一条完整流水线。

改动面V2 实际实现对应源码

01 — 项目概览

它到底解决了什么问题？

手动追踪一篇论文的数百个引用来源、判断引用者的学术影响力，是科研工作中极为耗时的任务。CitationClaw 将这一过程完全自动化。

🔍

自动化采集

通过 ScraperAPI 代理绕过 Google Scholar 的反爬机制，支持三级重试与地理节点轮换，稳定抓取所有引用论文。

🧠

结构化作者画像

优先通过 OpenAlex、Semantic Scholar、arXiv、WOS 与 PDF 提取作者和机构，再使用 Search LLM 对院士/Fellow/杰青等高影响力身份进行补充核验。

📊

自包含仪表盘

生成单个 HTML 文件，内嵌 Chart.js 交互图表、关键词词云、趋势预测、引用描述摘要，无需服务器即可打开分享。

02 — 设计哲学

为什么这样设计？

每一个技术决策的背后都有明确的工程理由。以下是驱动 CitationClaw 架构的核心原则。

01

OpenAI-compatible API 抽象

系统通过标准 openai.AsyncOpenAI / OpenAI SDK 与兼容 OpenAI Chat Completion 协议的服务对话。V2 将模型角色拆开：openai_model 应配置为具备搜索能力的模型，用于学者身份核验；dashboard_model 可配置为更便宜的轻量模型，用于报告生成、PDF 作者提取和引文语境抽取。

02

Async-first 并发架构

Phase 2 的结构化元数据查询、PDF 下载，以及 Phase 3/4 的 Search LLM 与轻量模型调用均为 I/O-bound 工作负载，使用 asyncio.Semaphore 控制并发数。Phase 1 中 ScraperAPI 的同步调用通过 asyncio.to_thread 包装，不阻塞事件循环。理论并发限制取决于 API 配额和外部服务速率限制，而非线程池大小。

03

三层重试弹性

Google Scholar 的反爬策略在三个维度上发力：网络层（HTTP 错误）、应用层（CAPTCHA/登录页）、基础设施层（数据中心地理不一致）。CitationClaw 针对每层单独设计对策，而非用一个简单的指数退避处理所有错误——这大幅提升了大规模抓取的成功率。

04

持久化增量缓存

作者信息缓存以 JSON 文件存储在 data/cache/author_info_cache.json，以论文 URL 或标题为 key，支持字段级增量更新。这意味着用不同配置重跑流水线时，已搜索的作者信息无需重新查询，极大节省 LLM API 费用。

05

自包含 HTML 输出

仪表盘的所有数据以 JS 变量形式 inline 在 HTML 中，Chart.js 和 marked.js 通过 CDN 加载，其余 CSS/JS 全部内嵌。这使得非技术用户可以直接双击打开，邮件附件发送，或上传至任何静态托管服务，而无需部署任何后端。

06

Python package + CLI 分发

v2 通过 pyproject.toml 暴露 citationclaw console script，CLI 内部启动 citationclaw.app.main:app 并自动打开浏览器。这样安装后的用户入口和开发者的 python -m citationclaw 入口指向同一套运行逻辑，避免维护额外启动脚本。

07

确定性降级 Fallback

Phase 5（仪表盘）的每个 LLM 分析步骤都有一个不依赖 LLM 的确定性算法作为后备：关键词分析用词频统计，趋势预测用线性回归（含同比增长率截断），洞察生成用模板字符串。这确保了即使 LLM API 完全不可用，系统仍能输出有意义的报告。

08

Skills Runtime — 模块化流水线

v2 将流水线阶段抽象为可注册的 Skill：每个 Skill 实现 Skill Protocol（含 name: str 和 async run(ctx, **kwargs) → SkillResult），由 SkillsRuntime 统一调度，通过 SkillContext 共享 config、log、progress、cancel_check 与 extras。相比 v1 的脚本式流程，v2 更容易替换单个阶段、做单元测试和定位运行问题。

02.5 — Skills Runtime（v2 架构核心）

模块化技能调度架构

v2 以 Skills Runtime 作为任务编排层，将业务阶段拆成可注册、可替换、可测试的执行单元。

🆕 核心组件

  • Skill Protocol — 定义 name: str + async run(ctx, **kwargs) → SkillResult 接口
  • SkillContext — 共享运行时上下文（config, log, progress, cancel_check, extras）
  • SkillResult — 统一结果信封（name + data dict）
  • SkillRegistry — 按名称注册/查找 Skill 的轻量容器
  • SkillsRuntime — 顶层调度器，build registry → resolve skill → build ctx → await run

🔎 设计亮点

零依赖接口

Skill Protocol 仅依赖标准库 dataclass + typing，任何实现了 name + run 的对象均可注册，无需继承基类。

上下文注入

SkillContext 持有 config、log callback、progress callback、cancel_check，Skill 无需直接访问全局状态，天然支持单元测试 mock。

统一 Registry

build_default_registry() 集中注册当前内置 Skill，可在测试或扩展时替换为自定义 registry，无需修改 runtime 逻辑。

⚠ 年份遍历的意义

Google Scholar 对任意查询最多返回约 1000 篇论文。对于高引用论文（如引用数超过 2000），不分年份直接抓取会丢失大量数据。年份遍历模式将问题分解为每年独立的子查询，每年通常不超过 300 篇，从而突破硬限制。

Phase 2

作者与元数据采集器

metadata_collector.py · structured APIs + PDF fallback

v2 不再把作者信息采集完全交给 LLM。当前主路径由 TaskExecutor._run_new_phase2_and_3() 执行：先把 Phase 1 的 page-based JSONL 展平成论文列表，按 link/title + canonical title 去重，再用 10 workers 并行查询结构化元数据。S2 是默认主源，OpenAlex 与 arXiv 作为补充；配置 WOS Key 时才启用 WOS → S2 → PDF 的结构化作者链。

async def collect(title, paper_url=""):
    # 1. S2 title search
    s2_result = await s2.search_paper(title)

    # 2. title miss 时尝试 S2 URL lookup

Phase 3

学者影响力评估与导出

scholar_prefilter.py · scholar_search_agent.py · exporter.py

Phase 3 只对非自引论文中的作者做影响力评估。系统先用 ScholarPreFilter 根据 h-index、总引用数和知名机构名单筛出候选作者，再按施引论文调用 ScholarSearchAgent.search_paper_authors()。Search LLM 的输入包含结构化姓名、机构、国家、h-index 和 citation_count，因此比 v1 只凭标题或作者字符串搜索更可控。

  • 红色 CAS/CAE 院士（中国两院院士）
  • 绿色 其他院士（外国院院士等）
  • 黄色 IEEE/ACM Fellow 等国际称号
  • 白色 / 普通：其他作者
  • 搜索结果写入 ScholarSearchCache，命中缓存时不再调用 Search LLM

主要 Excel 输出：

  • paper_results.xlsx — 完整施引论文与作者信息表
  • paper_results_all_renowned_scholar.xlsx — 所有知名学者引用汇总
  • paper_results_top-tier_scholar.xlsx — 顶尖学者子集
  • paper_results_with_citing_desc.xlsx — Phase 4 后生成的引文语境增强表
Phase 4

引文语境提取器

phase4_citation_extract.py · PDF parse + LLM review

v2 的 Phase 4 不再使用旧的 Web Search 摘要模块，而是复用 Phase 2 已下载的 PDF。PDFCitationParser 先把正文与参考文献分离，支持 [N] 和 (Author, Year) 两类引用格式，给段落打上章节标签并扩展上下文窗口；随后轻量模型执行 Extractor 提示词，必要时再由 Reviewer 提示词校验引用编号、参考文献条目和正文句子是否一致。

app/config_manager.py 负责读写 config.json，并通过 _safe_data_path() 对所有文件路径进行路径穿越攻击防护。

app/config_manager.py — 路径安全校验
Python
def _safe_data_path(relative_path: str) -> Path:
    """防止路径穿越攻击 (e.g. ../../etc/passwd)"""
    base = Path("data").resolve()
    target = (base / relative_path).resolve()
    if not str(target).startswith(str(base)):
        raise ValueError(f"Path traversal attempt detected: {relative_path}")
    return target
06 — 弹性设计

三层容错机制与缓存策略

三层重试架构

Google Scholar 的反爬机制在三个不同层面运作，系统针对每一层设计了独立的检测与对策逻辑：

T1
HTTP 层重试 — 网络错误与 5xx 响应
检测条件：HTTP 状态码 ≠ 200，或网络超时。
对策：轮换 ScraperAPI API Keys；按配置间隔（如 5,10,20 秒）指数退避；最多 N 次（默认 3）。
rotate api_keys
T2
CAPTCHA 层重试 — 登录页 / 验证码检测
检测条件：解析到的有效论文数 < 2（说明返回的是登录/验证码页面，而非真实结果）。
对策：轮换 ScraperAPI Session Number（生成新的随机 6 位 session ID），触发服务器重新分配代理 IP。
rotate session_number
T3
数据中心不一致层 — 短页面地理切换
检测条件：页面内容完整（非 CAPTCHA），但论文数量异常少（数据中心缓存不一致）。
对策：通过 ScraperAPI country_code 参数切换出口节点地理位置。前 20 次循环使用优先国家列表，之后随机抽取全球 60+ 国家池。
rotate country_code

10 — 已知限制与优化方向

诚实面对当前的权衡

每个工程决策都有代价。以下是当前版本的已知限制，以及有价值的改进方向。

🔁

Google Scholar 反爬机制持续演进

当前三层重试机制基于已观察到的规律，但 Scholar 的反爬策略随时可能更新。若 Scholar 引入新型检测机制（如行为分析、更复杂的 CAPTCHA），现有重试逻辑可能需要相应调整。改进方向：引入本地浏览器驱动（Playwright）作为终极 fallback，绕过网络层检测。

👤

学者影响力评估仍依赖 Search LLM 质量

v2 已通过 OpenAlex、Semantic Scholar、arXiv、WOS 与 PDF 提高作者元数据的确定性，但院士/Fellow/国家级人才等身份仍需要跨网页核验。对于同名学者或机构变更频繁的作者，Search LLM 仍可能混淆人物。改进方向：引入更细粒度的作者实体链接、官方名单比对和证据链接评分，进一步降低误判率。

⚠️

Advanced 的 Phase 4 范围过滤尚未在后端生效

配置和前端已经定义 citing_description_scope=renowned_only，但 execute_for_titles() 当前直接把 merged_authors.jsonl 传给 phase4_citation_extract，没有先过滤出知名学者相关论文。因此 Advanced 与 Full 在 Phase 4 覆盖范围上暂时等价。改进方向：在进入 Phase 4 前根据 Renowned Scholar / 指定学者名单生成 filtered JSONL，并把实际输入文件写入日志和报告元数据。

📈

Year-traverse 模式的时间开销

对于超高引用论文（5000+ 篇），年份遍历模式需要对每年分别发送多页请求，总体耗时可能达到数小时。当前无法中断并恢复（断点续抓）。改进方向：实现 JSONL append 模式 + 年份级别的进度检查点，支持中断恢复。

🔒

单用户架构：无认证与多用户隔离

当前系统设计为单用户本地工具，FastAPI 端点无认证保护，data/ 目录为全局共享。若部署到多用户环境，数据存在互相污染的风险。改进方向：引入任务 ID（UUID）隔离每次运行的数据，并添加 API Key 认证中间件。

🌝

仪表盘 LLM 调用为同步阻塞

Phase 5 的 4 次 LLM 调用使用 openai.OpenAI（同步客户端）按顺序执行，总等待时间约为 4 个 LLM 响应时间之和（可能 60-120 秒）。改进方向：迁移至 asyncio.gather() 并发执行，总时间缩减为最慢单次调用时间。

📅

缓存无过期策略

作者信息缓存一旦写入，永不自动过期。若作者获得新的学术头衔（如新当选院士），缓存中的旧数据不会自动更新。改进方向：为每个缓存条目添加 TTL（如 30 天），或提供 --invalidate-cache-before YYYY-MM-DD 参数。
